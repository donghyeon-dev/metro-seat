# 자리요 — 전략 문서

> **"지금 앉을래요?"** — 서서 가는 사람에게 가능성을 띄우는 앱.

---

## 0. 한 줄 정의
자리요는 지하철 하차 예정자가 자신의 자리를 자발적으로 공유하고, 서 있는 사람이 그 자리를 이어받을 수 있도록 돕는 PWA 서비스다. 좌석은 거래되지 않고 호의로 양도된다.

## 1. 북극성 (3개월 단위)
- **단기 (지금~3개월)**: MVP로 "실사용자가 실제로 매번 쓰는가"를 증명한다.
- **중기 (3~12개월)**: 특정 구간(예: 2호선 강남↔잠실 오전 7-9시)에서 주 5건 이상 자발적 매칭 유지.
- **장기 (12개월+)**: 수도권 출퇴근 피크 수만 명 규모, M1·M2 수익화로 인프라 비용 자가 충당.

## 2. 핵심 제약
- 1인 사이드 프로젝트, 주 10시간 내외
- 예산 ~0 (유료 획득 불가)
- 양면 네트워크 (Provider ↔ Seeker 동시 필요 → cold-start 리스크)
- 현재 프로덕션 준비도 ≈ 25%

## 4. 방향성 결정

### 포지셔닝 선언
> **"이 앱은 자리를 사고팔지 않습니다. 호의의 교환을 돕습니다."**

이 문장이 T5(법)·T4(PR) 방어의 뼈대이며, 이후 모든 수익·UI·약관이 이 선 위에서 설계된다.

### 전략 옵션 비교
| 옵션 | 요약 | 선택 |
|------|------|------|
| A. 마이크로 커뮤니티 침투 | 특정 구간·시간대 집중 증명 | ✅ **채택** |
| B. 감사 경제 선행 | 소액 송금 기반 Provider 유인 | 유보 (T5 전제 미준비) |
| C. 정보 공유 앱 피벗 | 혼잡도·빈자리 크라우드소싱 | 피벗 카드 (옵션 A 실패 시) |
| D. 공익 파트너십 선행 | 지자체·코레일 ESG 연계 | 장기 카드 (1인에 부적합) |

## 9. 백엔드 스택 관련 결정 (Non-Goal)

현재 스택: **Supabase(Managed) + Next.js API Routes(프록시)**. 비즈니스 로직의 상당 부분이 RLS·Postgres 트리거에 선언적으로 존재하므로 "실질 백엔드 코드"는 얇다.

### 당분간 하지 않는 것
- **Spring Boot·FastAPI·Nest/Express 등으로의 백엔드 이전**

### 이유 (1인·주10h 리소스 기준)
| 영역 | 이전 시 직접 구현해야 하는 것 | 추가 공수 |
|------|-----------------------------|----------|
| Auth | JWT 발급·refresh·세션 블랙리스트 | +1~2주 |
| Realtime | WebSocket 서버·채널·인증·스케일 | +2~4주 |
| RLS 대체 | 모든 쿼리에 권한 미들웨어 | +1~2주 |
| 자동 만료·트리거 | cron + 멱등 처리 | +3일 |
| 호스팅·관측 | 백엔드 호스팅·APM·tracing | +1주 |
| **총합** | | **6~10주** |

→ MVP 증명 일정 전체와 맞먹음.

### 이전을 재검토할 트리거
- WAU 5,000+ 도달 후 Supabase Realtime 비용이 월 예산 초과
- Supabase 벤더 락인·장애 리스크가 실제로 구현된 경우
- 매칭 알고리즘·정산 등 RLS로 표현 불가능한 로직 등장
- Supabase Edge Functions로도 해결 불가한 장시간 비동기 작업 필요

### 재검토 시 1순위
1. **Nest 또는 Express** (JS 단일 언어·타입 공유·마찰 최소)
2. **FastAPI** (Python 호환 필요 시)
3. Spring Boot는 1인 사이드에 오버엔지니어링 — 선택지에서 제외

### 현 리소스에서의 대체 경로
새 백엔드 없이 **Supabase Edge Functions(Deno)** 또는 **Vercel Functions**로 다음을 해결한다:
- 트롤 탐지 / 레이트 리밋 (DEV-BACKLOG T-3·T-4)
- 이벤트 수집·집계 (Q-1·Q-2)
- 3rd party Webhook 수신·정규화
